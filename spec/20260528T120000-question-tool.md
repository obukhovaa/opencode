# Question Tool — Interactive User Input

**Date**: 2026-05-28
**Status**: Implemented
**Author**: AI-assisted

## Overview

Add a `question` tool that lets agents ask the user structured questions during execution. The tool presents one or more questions — each with predefined options and an optional custom text input — and blocks until the user responds. In the TUI, questions appear as a modal dialog with selectable options (similar to the session dialog). In non-interactive mode (CLI `-p` flag), the tool is **not registered** — agents must not attempt to ask questions when running headless.

## Motivation

### Current State

When an agent needs user input mid-execution, it can only:
1. **Stop and respond with text** — ending the tool-use loop and requiring the user to re-prompt.
2. **Make assumptions** — which may be wrong and waste cycles.

There is no mechanism for the agent to pause execution, present structured choices, receive an answer, and continue the same tool-use loop.

### Problems

1. **Broken flow**: Agents must guess or stop when they encounter ambiguity, losing the execution context.
2. **No structured choices**: Even when the agent asks a question in text, the user's free-form reply must be re-parsed by the model — error-prone and expensive.
3. **Subagents cannot escalate**: A subagent that hits a decision point must either guess or fail — it cannot surface a question to the user.

### Desired State

```
Agent: I found 3 possible approaches for the database migration.
       [Question tool call with options: "Add column with default", "Create new table", "Alter existing column"]

User sees a modal dialog, picks "Add column with default"

Agent continues: Great, proceeding with ALTER TABLE ADD COLUMN with a default value...
```

The agent's tool-use loop is preserved — the question tool blocks, the user responds, and the agent continues with the answer in context.

## Design

### Reference Implementation

The design follows the opencode-dax TypeScript reference (`packages/opencode/src/question/` and `packages/opencode/src/tool/question.ts`), adapted to Go idioms:

- **Question service**: In-memory request/response coordination via channels (Go equivalent of Effect's `Deferred`)
- **Tool**: Thin wrapper that calls the service, formats answers into text
- **TUI dialog**: Modal with selectable options, custom text input, keyboard navigation
- **Non-interactive**: Tool not registered at all (not auto-answered)

### Key Differences from Reference

| Aspect | opencode-dax (TS) | This implementation (Go) |
|---|---|---|
| Concurrency primitive | Effect `Deferred` | Go channels |
| State management | `InstanceState` + `Map` | `sync.Map` with channel-based pending requests |
| Multi-question flow | Tabbed UI with confirmation tab | Sequential dialog progression (one question at a time, `(1/N)` counter) |
| Custom answer | "Type your own answer" as last option | Separate text input below options, always visible when enabled |
| HTTP API | REST endpoints for list/reply/reject | Not needed (TUI-only for now, extensible later) |

### Architecture

```
┌─────────────┐     ┌──────────────────┐     ┌──────────────┐
│ Question     │────▶│ Question Service │◀────│ TUI Dialog   │
│ Tool         │     │ (in-memory)      │     │ (modal)      │
│ (blocks)     │     │                  │     │              │
└─────────────┘     └──────────────────┘     └──────────────┘
                           │
                    pubsub.Broker
                           │
                    ┌──────┴───────┐
                    │ TUI app      │
                    │ subscribes   │
                    │ shows dialog │
                    └──────────────┘
```

## Detailed Design

### 1. Question Service (`internal/question/question.go`)

The service coordinates question requests between agents (via tool calls) and the UI layer.

```go
package question

type Option struct {
    Label       string `json:"label"`       // Display text (1-5 words)
    Description string `json:"description"` // Explanation of choice
}

type Prompt struct {
    Question string   `json:"question"` // Full question text
    Options  []Option `json:"options"`  // Available choices
    Multiple bool     `json:"multiple"` // Allow selecting multiple
    Custom   *bool    `json:"custom"`   // Allow custom answer (default: true)
}

type Request struct {
    ID        string   `json:"id"`
    SessionID string   `json:"session_id"`
    Questions []Prompt `json:"questions"`
}

type Service interface {
    pubsub.Suscriber[Request]
    // Ask blocks until the user responds or ctx is cancelled.
    // Returns one answer per question (each answer is a slice of selected labels).
    Ask(ctx context.Context, sessionID string, questions []Prompt) ([][]string, error)
    // Reply resolves a pending question request.
    Reply(requestID string, answers [][]string) error
    // Reject dismisses a pending question.
    Reject(requestID string) error
}
```

**Implementation details:**
- `Ask()` generates a UUID, stores a `chan [][]string` in a `sync.Map`, publishes the request via `pubsub.Broker`, and blocks on the channel.
- `Reply()` looks up the channel, sends answers, and removes the entry.
- `Reject()` sends a nil/error signal to unblock `Ask()`.
- On context cancellation, `Ask()` cleans up and returns `ErrQuestionRejected`.

### 2. Question Tool (`internal/llm/tools/question.go`)

```go
const QuestionToolName = "question"

type questionTool struct {
    service question.Service
}

func (q *questionTool) Info() ToolInfo {
    return ToolInfo{
        Name:        QuestionToolName,
        Description: questionDescription,
        Parameters: map[string]any{
            "type": "object",
            "properties": map[string]any{
                "questions": map[string]any{
                    "type": "array",
                    "description": "Questions to ask the user",
                    "items": map[string]any{
                        "type": "object",
                        "properties": map[string]any{
                            "question": map[string]any{
                                "type":        "string",
                                "description": "Complete question text",
                            },
                            "options": map[string]any{
                                "type":        "array",
                                "description": "Available choices",
                                "items": map[string]any{
                                    "type": "object",
                                    "properties": map[string]any{
                                        "label": map[string]any{
                                            "type":        "string",
                                            "description": "Display text (1-5 words, concise)",
                                        },
                                        "description": map[string]any{
                                            "type":        "string",
                                            "description": "Explanation of choice",
                                        },
                                    },
                                    "required": []string{"label", "description"},
                                },
                            },
                            "multiple": map[string]any{
                                "type":        "boolean",
                                "description": "Allow selecting multiple choices (default: false)",
                            },
                            "custom": map[string]any{
                                "type":        "boolean",
                                "description": "Allow typing a custom answer (default: true)",
                            },
                        },
                        "required": []string{"question", "options"},
                    },
                },
            },
            "required": []string{"questions"},
        },
        Required: []string{"questions"},
    }
}
```

**Tool behavior:**
- Extracts `sessionID` from context
- Calls `service.Ask(ctx, sessionID, questions)`
- Formats answers: `"question_text"="answer1, answer2"` pairs
- Returns: `User has answered your questions: ...`
- On rejection: Returns error response `"The user dismissed the question"`
- `AllowParallelism`: always `false` (blocks on user input)
- `IsBaseline`: `true`

### 3. Tool Description (`internal/llm/tools/question.go`, embedded)

```
Use this tool when you need to ask the user questions during execution. This allows you to:
1. Gather user preferences or requirements
2. Clarify ambiguous instructions
3. Get decisions on implementation choices as you work
4. Offer choices to the user about what direction to take

Usage notes:
- When `custom` is enabled (default), a "Type your own answer" option is added automatically; don't include "Other" or catch-all options
- Answers are returned as arrays of labels; set `multiple: true` to allow selecting more than one
- If you recommend a specific option, make that the first option in the list and add "(Recommended)" at the end of the label
```

### 4. Agent Integration (`internal/llm/agent/tools.go`)

The question tool is a **manager tool** — available only to primary agents (mode=`agent`), never to subagents.

```go
var managerToolNames = []string{
    TaskToolName,
    tools.QuestionToolName,    // ← add here
    tools.CronCreateToolName,
    // ...
}
```

**Non-interactive gating**: The question tool requires the `question.Service` to be initialized — which only happens when the app runs in interactive (TUI) mode. In non-interactive mode (`-p` flag), `app.Questions` is nil, so the factory returns nil and the tool is not registered.

In `NewToolSet`, the `createTool` function:
```go
case tools.QuestionToolName:
    if qSvc := factory.QuestionService(); qSvc != nil {
        return tools.NewQuestionTool(qSvc)
    }
    return nil
```

The `AgentFactory` interface gets two new methods following the setter injection pattern (matching `SetCronServices`/`CronServices` and `SetTodoStore`/`TodoStore`):
```go
type AgentFactory interface {
    // ... existing methods
    SetQuestionService(svc question.Service)
    QuestionService() question.Service
}
```

In `cmd/root.go`, the question service is only created for interactive mode:
```go
// Interactive mode — before TUI setup
questionSvc := question.NewService()
app.Questions = questionSvc
app.AgentFactory.SetQuestionService(questionSvc)

// Non-interactive mode — app.Questions stays nil, factory returns nil
```

### 5. TUI Question Dialog (`internal/tui/components/dialog/question.go`)

A modal dialog similar to the session dialog, but specialized for question answering.

**Interface:**
```go
type QuestionResponseMsg struct {
    RequestID string
    Answers   [][]string  // one answer per question
    Rejected  bool
}

type QuestionDialogCmp interface {
    tea.Model
    layout.Bindings
    SetQuestion(request question.Request) tea.Cmd
}
```

**UI layout:**
```
┌─ Question ──────────────────────────────────┐
│                                             │
│  How should we handle the migration?        │
│                                             │
│  ● Add column with default (Recommended)    │
│    Adds a new column with DEFAULT value     │
│                                             │
│    Create new table                         │
│    Creates a separate table for new data    │
│                                             │
│    Alter existing column                    │
│    Modifies the existing column in-place    │
│                                             │
│  ┌─────────────────────────────────────┐    │
│  │ Or type your own answer...          │    │
│  └─────────────────────────────────────┘    │
│                                             │
│  ↑↓ navigate  ⏎ select  esc dismiss        │
└─────────────────────────────────────────────┘
```

**Behavior:**
- **Single select** (`multiple: false`, default): Pressing Enter on an option immediately submits that answer.
- **Multi select** (`multiple: true`): Space toggles selection, Enter submits all selected.
- **Custom answer**: A text input at the bottom. When focused and non-empty, Enter submits the typed text as the answer.
- **Option rendering**: Each option shows `label` on the first line (highlighted when selected) and `description` indented below in muted text. Long labels/descriptions wrap to fit the dialog width.
- **Scrolling**: When options exceed the visible area, the list scrolls with the selection (like the session dialog).
- **Dismiss**: Escape rejects the question — the tool returns an error to the agent.

**Key bindings:**
| Key | Action |
|---|---|
| `↑` / `k` | Move selection up |
| `↓` / `j` | Move selection down |
| `Enter` | Submit selected option(s) / custom text |
| `Space` | Toggle selection (multi-select mode) |
| `Tab` | Switch focus between options list and custom input |
| `Esc` | Dismiss (reject question) |

**Multi-question handling**: Each tool call can contain multiple questions. The dialog processes them **one at a time** — showing the current question with a `(1/3)` counter in the title. After answering question N, the dialog advances to question N+1. After the last question, all answers are submitted together.

### 6. App Integration

**`internal/app/app.go`:**
```go
type App struct {
    // ... existing fields
    Questions question.Service  // nil in non-interactive mode
}
```

**`cmd/root.go` — interactive mode setup:**
```go
// Before TUI setup
app.Questions = question.NewService()

// In setupSubscriptions, add question subscription
questionCh := app.Questions.Subscribe(pubsub.CreatedEvent)
```

**TUI subscription handling (in the TUI event loop):**
When a question request arrives via `questionCh`, send a `ShowQuestionMsg` to the TUI program, which triggers the question dialog.

### 7. Non-Interactive Mode & Server Mode

By default, in non-interactive mode (`-p` flag), server mode (`serve`), and ACP mode, the question service is **not created** (`app.Questions` remains nil). The `AgentFactory.QuestionService()` returns nil, so `NewToolSet` skips the question tool entirely.

**Opt-in via env var**: Setting `OPENCODE_ENABLE_QUESTION_TOOL=1` enables the question tool in any mode. This is useful for:
- **Server mode** (`opencode serve`): External UIs can interact with questions via HTTP API
- **ACP mode** (`opencode acp`): IDE clients that support interactive prompts
- **Non-interactive mode** (`-p`): Scripting scenarios with external answer providers

### 8. HTTP API for Questions

When the question service is active (TUI mode always, other modes with env flag), three REST endpoints are available:

| Method | Endpoint | Description |
|---|---|---|
| `GET /question` | List pending questions | Returns pending question requests |
| `POST /question/{requestID}/reply` | Reply to question | Body: `{"answers": [["label1"], ["label2"]]}` |
| `POST /question/{requestID}/reject` | Reject question | Dismisses the question |

Question events are also streamed via SSE on `GET /event` as `question.asked` events, matching the `permission.asked` pattern.

## Edge Cases

1. **Agent cancelled while question pending**: Context cancellation propagates through `Ask()`, which cleans up the pending request. The dialog (if shown) is dismissed.
2. **Multiple questions from same agent**: Each tool call blocks independently. The agent's tool-use loop is single-threaded per call, so only one question can be pending per agent at a time.
3. **Session switch while question shown**: The dialog should be dismissed (rejected) if the user switches sessions.
4. **Empty options list**: When `custom` is enabled (default) and options are empty, show only the text input. When `custom` is disabled and options are empty, return a validation error to the agent.
5. **Dialog vs permission overlay**: The question dialog and permission dialog cannot appear simultaneously — the question tool blocks the agent loop, preventing further tool calls (and thus permission requests) while the question is pending. No special handling needed.

## Security Considerations

- The question tool does not execute any system commands or modify files — it's purely informational.
- No permission check needed (the tool only blocks and returns user text).
- Custom text input is returned as-is to the agent — the agent processes it in its next turn, subject to normal safety guardrails.

## Future Extensions

- **Default answers**: Allow specifying a default answer that auto-submits after a timeout.
- **`--question-file` flag**: Provide pre-computed answers from a JSON file for scripting scenarios.
- **Polling endpoint**: Maintain a list of pending questions in `GET /question` for clients that miss SSE events.
