# Session Recap (Away Summary)

**Date**: 2026-04-28
**Status**: Proposed
**Author**: AI-assisted

## Overview

Add a persistent "session recap" feature that generates a short 1-3 sentence summary of where a session left off. The recap is displayed at the bottom of the chat (after the last message) when a user resumes an old session, giving them instant context right where their viewport lands.

Inspired by Claude Code's `awaySummary` feature, but adapted for OpenCode's persistent session model — recaps are stored in the database and survive process restarts.

## Motivation

### Current State

When a user opens OpenCode and picks an old session from the session dialog, they see the full message history and must scroll/read to remember what they were doing. There is no "here's where you left off" affordance.

### Desired State

When resuming a session that has existing messages, a short recap appears at the top of the chat feed. The recap:
- Is generated lazily on session resume (not at process exit, which risks blocking SIGTERM)
- Is persisted so it doesn't need regeneration on every resume
- Is completely asynchronous — never blocks user input or agent interaction
- Is invisible to the LLM (human-only UI element)
- Does not appear in non-interactive mode (prompt / flow)

## Research Findings

### Claude Code Implementation

Claude Code's `awaySummary.ts` generates a recap when the terminal regains focus after 5+ minutes of blur. Key characteristics:
- **Transient** — not persisted, lives only in current process memory
- **Human-only** — filtered out before sending messages to the LLM
- **Trigger**: terminal focus events (DECSET 1004), 5-minute blur delay
- **Generation**: small/fast model, last 30 messages + session memory as context
- **Display**: inline system message in chat feed, dimmed styling

### OpenCode Differences

| Aspect | Claude Code | OpenCode |
|---|---|---|
| Session persistence | In-memory, single process | SQLite/MySQL, multi-process |
| Session resume | Same terminal refocus | Session picker dialog, process restart |
| Mode detection | Always TUI | TUI vs non-interactive (prompt/flow) |
| Summarizer | Dedicated small model call | Existing `summarizer` agent with provider |

### Existing Infrastructure

1. **Summarizer agent** (`config.AgentSummarizer`): Already used for context compaction. Has a dedicated provider created in `agent.go:122`. Uses `prompts/compaction.md` as the prompt template.

2. **Session model** (`internal/session/session.go`): Has `SummaryMessageID` for compaction summaries. Sessions support `ON DELETE CASCADE` for child records.

3. **Message model** (`internal/message/message.go`): Messages have `Role` (User/Assistant/System/Tool) and `Parts` (typed content parts). Stored with session foreign key + cascade delete.

4. **Langfuse tracing** (`internal/langfuse/`): Standalone traces are created via `TraceStart()` with `IsChild=false`. Title generation already follows this pattern — fire-and-forget goroutine with its own trace.

5. **Non-interactive detection** (`cmd/root.go`): `-p`/`--prompt` and `-F`/`--flow` flags trigger non-interactive code paths. TUI is the default when neither flag is set.

6. **TUI chat rendering** (`internal/tui/components/chat/`): Messages are rendered as `uiMessage` structs. Assistant messages already have special rendering for summary messages (tagged with "(summary)" label).

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Storage | Dedicated `session_recaps` table | Recaps are not conversation messages — they shouldn't appear in message lists, affect message counts, or be sent to the LLM. A separate table with `ON DELETE CASCADE` on `session_id` keeps them cleanly isolated. |
| Generation trigger | On session resume, if no current recap exists | Generating at process exit risks blocking SIGTERM. Generating on resume is lazy — only sessions that are actually revisited pay the cost. |
| Staleness | Regenerate if session's `max_seq` (from messages table) differs from recap's snapshot | `session.message_count` is maintained by DB triggers which may be missing on some MySQL installations. Instead, we query `MAX(seq)` from the messages table directly — it's monotonically increasing, always accurate, and uses an indexed column. The recap's `message_count` column stores this `max_seq` value at generation time. |
| Blocking behavior | Fully async — goroutine, never blocks UI or agent | The user can start typing immediately. The recap appears when ready (or on next resume if generation is slow). |
| Model visibility | Human-only, never injected into LLM context | The LLM already has the full message history (or compacted summary). Injecting the recap would waste tokens on redundant info. |
| Non-interactive mode | Skip entirely | No TUI = no user looking at the screen. Recaps are a visual affordance. |
| Message window | Last 30 messages (configurable constant) | Matches Claude Code's strategy. 30 messages ≈ 15 exchanges — enough context for "where we left off" without hitting token limits on the summarizer model. |
| Agent reuse | Use existing `summarizer` agent's provider | No new model configuration needed. The summarizer is already configured and its provider is created for every primary agent. Recap generation reuses `summarizeProvider` with a different prompt. |
| Langfuse attribution | Standalone trace, name derived from `AgentIDContextKey` = `"summarizer"` | Same pattern as title generation — independent trace, not nested under an agent turn. Trace name is `"summarizer"` (derived by `createLangfuseTrace` from the agent ID context value). Tagged with session ID for correlation. |
| Prompt design | Short, focused on "what + next step" | Adapted from Claude Code: "State the high-level task, then the concrete next step. Skip status reports and commit recaps." |
| Display | Dimmed styled block at bottom of chat, after last message | Visually distinct from conversation messages. Shows a "◆" marker. Placed after the last message so it's immediately visible without scrolling (viewport is at the bottom on resume). Cleared from the *display* when the user sends a new message (the recap has served its purpose). The persisted recap remains in the DB — it will be shown again if the session is resumed later, or regenerated if stale. |

## Architecture

### Database Schema

New table `session_recaps`:

```sql
CREATE TABLE IF NOT EXISTS session_recaps (
    id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL UNIQUE,
    content TEXT NOT NULL,
    message_count INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL,
    FOREIGN KEY (session_id) REFERENCES sessions (id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_session_recaps_session_id ON session_recaps (session_id);
```

- `UNIQUE` on `session_id` — one recap per session, upserted on regeneration.
- `ON DELETE CASCADE` — recap is automatically deleted when session is deleted.
- `message_count` — snapshot of `sessions.message_count` at generation time, used as staleness signal (compare against current session message count on resume).
- No `updated_at` — recaps are upserted in place via `ON CONFLICT ... DO UPDATE`.

### Recap Service

New package `internal/recap/`:

```go
type Recap struct {
    ID           string
    SessionID    string
    Content      string
    MessageCount int64
    CreatedAt    int64
}

type Service interface {
    // Get returns the recap for a session, or empty Recap if none exists.
    Get(ctx context.Context, sessionID string) (Recap, error)
    // Save creates or replaces the recap for a session.
    // messageCount is the session's message_count at generation time (staleness signal).
    Save(ctx context.Context, sessionID string, content string, messageCount int64) (Recap, error)
    // Delete removes the recap for a session.
    Delete(ctx context.Context, sessionID string) error
}
```

### Generation Flow

```
User resumes session (TUI session dialog → SessionSelectedMsg)
    │
    ├── Guard: session.MessageCount == 0 → no-op (empty/new session, including first launch)
    │
    ├── TUI loads messages, renders chat as usual (no blocking)
    │
    └── TUI checks: does this session have a current recap?
        │
        ├── YES, recap.MessageCount == session.MessageCount
        │   └── Display existing recap at top of chat (fresh)
        │
        ├── YES, but recap.MessageCount != session.MessageCount (stale)
        │   ├── Display stale recap immediately (better than nothing)
        │   └── Fire goroutine to regenerate → update display when done
        │
        └── NO recap exists
            └── Fire goroutine to generate → display when ready
```

### Generation Logic

```go
const RecentMessageWindow = 30

func generateRecap(ctx context.Context, summarizeProvider provider.Provider, messages []message.Message) (string, error) {
    if len(messages) == 0 {
        return "", nil
    }

    recent := messages
    if len(recent) > RecentMessageWindow {
        recent = recent[len(recent)-RecentMessageWindow:]
    }

    // Append the recap prompt as a user message
    recent = append(recent, message.Message{
        Role: message.User,
        Parts: []message.ContentPart{
            message.TextContent{Text: recapPrompt},
        },
    })

    response, err := summarizeProvider.SendMessages(ctx, recent, nil)
    if err != nil {
        return "", err
    }

    return strings.TrimSpace(response.Content), nil
}
```

### Recap Prompt

```
The user stepped away and is coming back to this session.
Write exactly 1-3 short sentences to remind them where they left off.
Start by stating the high-level task — what they are building, debugging, or investigating.
Then state the concrete next step or last action taken.
Skip status reports, file lists, and commit recaps.
Be direct and specific.
```

### Langfuse Integration

Recap generation creates a standalone trace, following the same pattern as title generation in `agent.go:200-214`:

```go
ctx = context.WithValue(ctx, tools.SessionIDContextKey, sessionID)
ctx = context.WithValue(ctx, tools.AgentIDContextKey, config.AgentName("summarizer"))
ctx = a.createLangfuseTrace(ctx, sess)
defer langfuse.EndTrace(ctx)
```

Trace name will be `"summarizer"` — derived by `createLangfuseTrace` from `AgentIDContextKey`. This reuses the existing agent ID rather than introducing a special-case override.

### TUI Integration

#### Display

The recap is rendered as a styled block at the top of the viewport, before the first message. It uses dimmed/muted text colors from the theme to visually separate it from conversation messages.

```
 ◆ You were adding a session recap feature to OpenCode. Next step: implement
   the TUI rendering for the recap block.
```

The recap block is:
- Rendered by the `messagesCmp` component in `chat/list.go`
- Stored as a field on the component (not in the messages slice)
- Cleared from *display only* when the user sends a new message in the session — the persisted recap remains in the DB and will be shown again on next resume (or regenerated if stale by then)

#### Async Update

When a recap is generated asynchronously, the TUI receives a `tea.Msg`:

```go
type RecapReadyMsg struct {
    SessionID    string
    Content      string
    MessageCount int64 // snapshot from generation time, used for Save()
}
```

The `messagesCmp` handles this message by updating the recap field and re-rendering. The `MessageCount` is captured at generation start (before calling `SendMessages`) so that the staleness snapshot reflects the state the recap was generated from, not the state at persist time.

### Agent Integration

The recap generation method is added to the agent `Service` interface:

```go
type Service interface {
    // ... existing methods ...
    GenerateRecap(ctx context.Context, sessionID string) (string, error)
}
```

This method:
1. Loads the last N messages for the session
2. Calls `summarizeProvider.SendMessages()` with the recap prompt
3. Returns the generated text (caller persists it)

The TUI orchestrates the full flow: check recap service → call agent → persist via recap service → send `RecapReadyMsg`.

## Edge Cases

1. **Session with 0 messages**: Skip recap generation entirely.
2. **Session with only 1-2 messages**: Generate anyway — the summarizer handles short conversations gracefully.
3. **Concurrent recap generation**: The `UNIQUE` constraint on `session_id` prevents duplicates. If two processes try to generate simultaneously, the last writer wins (acceptable — both produce similar content).
4. **Summarizer provider unavailable**: `summarizeProvider` is nil when the agent is a subagent (mode != agent). Recap generation should only be triggered from the primary agent's TUI, where the provider exists.
5. **Very long sessions**: The 30-message window caps input size. Even with compacted sessions (where older messages are replaced by a summary), the window captures the most recent context.
6. **Process exit during generation**: The goroutine is cancelled. No recap is persisted. It will be generated on next resume — this is by design.
7. **Rapid session switching**: Cancel any in-flight recap generation for the previous session before starting generation for the new one.

## Success Criteria

- [ ] Resuming an old session shows a recap within 2-3 seconds (async, non-blocking)
- [ ] Recap is persisted and survives process restart
- [ ] Recap is not visible to the LLM (not in message history)
- [ ] Deleting a session cascades to delete its recap
