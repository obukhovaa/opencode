# TodoWrite Tool

**Date**: 2026-05-17
**Status**: Implemented

## Overview

Add a `todowrite` tool that lets primary agents (mode=`agent`) maintain a structured task list during long-running sessions. The tool is write-only from the agent's perspective ‚Äî it replaces the entire list on each call. External UIs (OpenWork, AionUI) read the list via `GET /session/{id}/todo` and ACP `session/update` notifications.

## Motivation

Long sessions with 10+ tool calls lose track of multi-step work. Both Claude Code and dax opencode solve this with a `todowrite` tool:

1. The agent calls `todowrite` with the full task list whenever progress changes
2. The system prompt guides the agent to update proactively
3. External UIs display a progress panel with checkmarks

Our fork lacks this. The existing `task` tool spawns subagents ‚Äî it doesn't track progress within a session. The `todowrite` tool is complementary: it tracks what the *current* agent is doing, not what subagents are doing.

### What we're NOT building

- A todo "read" tool ‚Äî the system prompt instructs the agent to keep updating proactively
- Database persistence ‚Äî todos are session-scoped and transient; in-memory is sufficient
- Subagent todos ‚Äî subagents receive focused single tasks and don't benefit from a todo list
- Periodic reminders ‚Äî dax opencode doesn't use them; the system prompt is sufficient to guide the agent

## Design

### Data model

```go
// internal/todo/todo.go
type Item struct {
    Content  string `json:"content"`   // Brief task description
    Status   string `json:"status"`    // pending, in_progress, completed, cancelled
    Priority string `json:"priority"`  // high, medium, low
}
```

Status transitions: `pending` ‚Üí `in_progress` ‚Üí `completed` or `cancelled`.

### Storage

In-memory map keyed by session ID, protected by a `sync.RWMutex`. The store lives on `*app.App` so it's accessible from the tool, the HTTP server, and the ACP handler.

```go
// internal/todo/store.go
type Store struct {
    mu    sync.RWMutex
    items map[string][]Item // sessionID -> items
}

func (s *Store) Set(sessionID string, items []Item)
func (s *Store) Get(sessionID string) []Item
func (s *Store) Delete(sessionID string)
```

The store is created in `app.New()` and stored as `app.Todos *todo.Store`. Session deletion should call `store.Delete(sessionID)` to avoid leaking memory.

### Tool implementation

**File**: `internal/llm/tools/todowrite.go`

```
Tool name:    todowrite
Permission:   allow (no permission prompt ‚Äî it's metadata, not a side effect)
Parallelism:  false (always sequential ‚Äî each call replaces the full list)
```

**Parameters**:

| Field | Type | Description |
|-------|------|-------------|
| `todos` | `[]Item` | The complete, updated todo list |

Each `Item`:

| Field | Type | Description |
|-------|------|-------------|
| `content` | string | Brief task description |
| `status` | string | `pending`, `in_progress`, `completed`, `cancelled` |
| `priority` | string | `high`, `medium`, `low` |

**Behaviour**:

1. Parse the `todos` array from input JSON
2. Store the list via `store.Set(sessionID, todos)`
3. Count remaining (non-completed, non-cancelled) tasks
4. Return a response:
   - Content: `"Todos updated. N remaining. Continue with current tasks."`
   - Metadata: JSON object `{"todos": [...]}` ‚Äî used by TUI and converters to render the list

**Tool result metadata** is the key rendering signal. The TUI and API converters read `metadata.todos` to display the checklist with strikethrough for completed items.

### Tool registration

The `todowrite` tool is added to the tool set for agents with `mode=agent` only. It follows the same pattern as cron tools:

- In `internal/llm/agent/tools.go` `NewToolSet()`, add `todowrite` alongside existing tools
- The tool is gated on agent mode: only `agent` mode gets it, `subagent` does not
- The tool is enabled by default (no `tools: {"todowrite": false}` needed)
- Agents can disable it via config: `"tools": {"todowrite": false}`

### System prompt addition

Add to the agent prompt (same location as `cronToolPrompt` in `internal/llm/prompt/prompt.go`). This matches the dax opencode prompt ‚Äî no periodic reminders needed; the prompt itself guides the agent to update proactively:

```
# Task Tracking

Create and maintain a structured task list for the current coding session. Tracks progress, organizes multi-step work, and surfaces status to the user.

## When to use
Use proactively when:
- The task requires 3+ distinct steps or actions (not just 3 tool calls for a single conceptual step)
- The work is non-trivial and benefits from planning
- The user provides multiple tasks (numbered or comma-separated) or explicitly asks for a todo list
- New instructions arrive - capture them as todos
- You start a task - mark it `in_progress` (only one at a time) before working
- You finish a task - mark it `completed` and add any follow-ups discovered during the work

## When NOT to use
Skip when:
- The work is a single, straightforward task (or <3 trivial steps)
- The request is purely informational or conversational
- Tracking adds no organizational value

## States
- `pending` - not started
- `in_progress` - actively working (exactly ONE at a time)
- `completed` - finished successfully
- `cancelled` - no longer needed

## Rules
- Update status in real time; don't batch completions
- Mark `completed` only after the required work is actually done, including any required verification. Never based on intent.
- Keep exactly one `in_progress` while work remains
- If blocked or partial, keep it `in_progress` and add a follow-up todo describing the blocker
- Preserve user-provided commands verbatim (flags, args, order)
- Items should be specific and actionable; break large work into smaller steps
- Each call replaces the entire list ‚Äî always include all items
```

**Design rationale**: dax opencode does not implement periodic automatic reminders for todos. The system prompt is sufficient ‚Äî the agent follows the guidance voluntarily. This avoids complexity in the agent loop, avoids LLM cache invalidation (transient injections would break prefix caching), and matches the upstream implementation.

### HTTP endpoint

**`GET /session/{sessionID}/todo`** ‚Äî already stubbed, needs real implementation.

Change from returning `[]` to returning `app.Todos.Get(sessionID)`:

```go
func (s *Server) handleSessionTodo(w http.ResponseWriter, r *http.Request) {
    sessionID := r.PathValue("sessionID")
    todos := s.app.Todos.Get(sessionID)
    writeJSON(w, http.StatusOK, todos)
}
```

### ACP notification

When the todowrite tool runs, the ACP handler's event loop already receives the message update with the tool result. The dax ACP agent specifically looks for `todowrite` tool calls and emits a `session/update` with `sessionUpdate: "plan"`:

```json
{
  "sessionUpdate": "plan",
  "entries": [
    {"priority": "medium", "status": "in_progress", "content": "Implement handler"},
    {"priority": "medium", "status": "pending", "content": "Write tests"}
  ]
}
```

Add this to `internal/acp/handler.go` in `handleMessageEvent()`: when a `ToolResult` with `Name == "todowrite"` is received, parse the metadata and emit a plan update notification.

### SSE event

The existing SSE handler already emits `message.updated` events which include tool results. No additional SSE event type is needed ‚Äî OpenWork reads the todo list via the HTTP endpoint, not SSE.

### TUI rendering

The TUI already renders tool call metadata via the chat message list. The `todowrite` tool result will show as a tool call with the metadata rendered. The metadata `{"todos": [...]}` should be rendered as a checklist:

```
todowrite
  ‚úì Set up project structure
  ‚Üí Implement HTTP handler
  ‚óã Write tests
  ‚óã Update docs
```

Where `‚úì` = completed, `‚Üí` = in_progress, `‚óã` = pending, `~` = cancelled.

This rendering logic goes in the TUI's tool call rendering code. It checks if the tool name is `todowrite` and the metadata contains a `todos` array, then renders the checklist instead of raw JSON.

## Edge Cases

- **Agent calls todowrite with empty list**: Store clears the list, tool returns "No todos."
- **Session compaction**: Todos survive compaction because they're in-memory, not in message history.
- **Multiple agents on same session**: Each agent mode shares the same session's todo list. This is intentional ‚Äî if a hivemind agent delegates and returns, the coder agent should see the same todos.
- **Memory growth**: Todos are small (few KB per session). Deleted sessions clean up via `store.Delete()`. No unbounded growth concern.

## Compatibility

- **dax opencode SDK**: The `GET /session/{id}/todo` response format matches ‚Äî array of `{content, status, priority}` objects
- **OpenWork**: The snapshot endpoint already calls this and renders the `TodoPanel` component
- **ACP/AionUI**: The `plan` session update notification matches the dax ACP agent's format
- **Backward compatible**: New tool, no changes to existing tools or APIs
