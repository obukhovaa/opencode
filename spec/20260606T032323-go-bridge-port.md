# Port openwork-router Bridge to Go (in-process)

**Date**: 2026-06-06
**Status**: Draft
**Author**: AI-assisted

## Overview

Port the openwork-router TypeScript bridge (`apps/opencode-router` in the openwork repo) into this Go fork as an in-process subsystem under `internal/bridge/`. Eliminates the second Node runtime, the HTTP loopback between bridge and opencode, and the cross-repo coordination cost for cross-cutting features.

## Motivation

### Current State

Two processes, two languages, two deploy artifacts:

```
┌──── opencode (Go) ────┐   loopback HTTP / SSE   ┌──── opencode-router (TS) ────┐
│ serve --port 3456     │ ◄─────────────────────► │ Node.js + npm deps           │
│ HTTP/SSE API, agents, │                         │ grammy / slack-sdk / mm-ws   │
│ sessions, sqlite      │                         │ own sqlite (sessions, binds) │
└───────────────────────┘                         └──────────────────────────────┘
```

Bridge size: **~7.6k LOC TS** (`bridge.ts` 3010, `mattermost.ts` 649, `health.ts` 806, `telegram.ts` 442, `slack.ts` 459, `db.ts` 260, plus ~1.5k LOC of helpers/types) plus **~3.3k LOC of tests** across 11 test files. **50 MB `node_modules` / 71 npm packages**.

Recent friction symptoms:

1. **Cross-cutting features require coordinated PRs in two repos**. The last three changes (message-part SSE events, agent/model selection endpoints, FILE: reply convention) each needed paired commits in both repos. Review and rollout are serialized.
2. **HTTP loopback for things that don't need to be HTTP**. The bridge subscribes to opencode's SSE event stream, parses JSON, and forwards tool transitions to chat — all on a localhost socket. The same data is a Go channel away on a `pubsub.Broker` inside opencode itself.
3. **Node runtime drift**. 71 npm packages have their own advisory churn; the bridge's `dist/` build artifact must be regenerated for every source change; the `bun:sqlite` vs `better-sqlite3` choice has bitten us with platform-specific binaries.
4. **Two SQLite databases**. opencode owns sessions/messages/permissions/questions. The bridge owns its own `~/.openwork/opencode-router/opencode-router.db` with peer bindings and session pointers. Two backup stories, two migration stories.
5. **Failure mode visibility**. When opencode restarts, the bridge logs `event stream closed` and reconnects. When the bridge crashes, no one notices unless the user pings the bot. Embedded ops would make the bridge a first-class concern of the opencode process.

The TS bridge is well-factored and works. Nothing about this port is fixing bugs — it's removing a redundant integration boundary.

### Desired State

Single binary, single language, single database, single config file:

```
┌──── opencode (Go) ───────────────────────────────────────────────┐
│ serve --port 3456    reads .opencode.json `router` section       │
│                                                                  │
│ ┌─ existing ─────────────────────┐  ┌─ new: internal/bridge ──┐  │
│ │ API, agents, sessions,         │  │ telegram, slack, mm     │  │
│ │ messages, pubsub.Broker        │◄─┤ adapters; orchestrator; │  │
│ │ (PartEvent, etc), sqlite|mysql │  │ chat commands; /send    │  │
│ └────────────────────────────────┘  └─────────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

Operationally: add a `router` section to `.opencode.json` with channels + tokens, run `opencode serve`. No second process. No second config file. No npm. No HTTP loopback.

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Where the code lives | New `internal/bridge/` package, with sub-packages per adapter (`telegram`, `slack`, `mattermost`) | Mirrors existing layout (`internal/api`, `internal/llm/agent`); keeps bridge isolated from core opencode |
| How it's launched | `opencode serve` reads the `router` section of `.opencode.json` and starts the bridge if at least one channel is configured | Single binary, single config file, simplest ops |
| Config storage | Embed under `.opencode.json` → `router: {channels, questionMode, permissionMode, toolUpdatesEnabled}`. **No** separate `opencode-router.json` | One config file, one mental model. HTTP mutations (identity CRUD, groups toggle) write back via `config.UpdateCfgFile` (today: package-private `updateCfgFile` at `internal/config/config.go:1205` — needs export and a temp-write+rename upgrade; see Configuration section) |
| Workspace directory | Pinned to `config.WorkingDirectory()`. **No** per-identity `directory` field, **no** per-peer binding row, **no** `/dir` chat command | One opencode process = one workspace. The whole "binding" abstraction in the TS bridge was a workaround for a separate bridge process serving multiple opencodes — irrelevant here |
| Bridge file storage | Subdirectory under `config.Data.Directory` (e.g. `<Data.Directory>/bridge/media/`) | Reuses existing data-dir convention; no new data paths to back up |
| HTTP API surface | Reuse opencode's existing mux + middleware for `/send`, identity CRUD, groups toggle | No second HTTP server in the process |
| Internal calls (sessions, prompts, events) | Direct Go calls into `internal/session`, `internal/message`, `internal/llm/agent` services. **No HTTP loopback** | The whole reason we're doing this. Removes the SSE serialize/parse round-trip and the "is opencode reachable" failure mode |
| Storage backend | Reuse opencode's existing `internal/db` (SQLite + MySQL, both first-class via sqlc-generated queriers and parallel goose migrations) and add two bridge tables (`bridge_sessions`, `bridge_allowlist`) on both providers, with `project_id` in every PK for MySQL multi-tenant safety | One database, one migration story, same provider abstraction. `bridge_bindings` and `bridge_settings` are not needed — directory is fixed, settings live in `.opencode.json` |
| Data migration from TS bridge | **None.** Fresh schema, no production traffic to preserve | Trims scope significantly. No importer, no cutover plan, no row-copy tool |
| File locking (single-writer election on SQLite) | Hand-rolled in `internal/fileutil/lock.go` — wrap `syscall.Flock` on Unix and `LockFileEx` via `golang.org/x/sys/windows` with `//go:build` tags | ~30 LOC per OS; not worth a third-party dep |
| Telegram lib | `go-telegram/bot` | Cleaner API, sufficient feature set |
| Slack lib | `slack-go/slack` | Only viable choice with Socket Mode |
| Mattermost lib | `mattermost/mattermost/server/public/model` | Official; framing/auth provided. Reconnect loop still hand-rolled |
| CLI subcommands (`telegram add`, etc.) | Drop. Mutations go through the HTTP API or direct `.opencode.json` edits | Removes 763 LOC of CLI code; dashboard / HTTP are the canonical mutation paths |
| `OPENCODE_ROUTER_MODEL` env override | Drop. Use `/model <id>` chat command | One way to set model. The env var was a workaround for the missing API |
| Per-peer model overrides | Already dropped in `780446a9`/`6878ece6` | Done |
| TS bridge retirement | Mark deprecated in openwork repo once Go bridge ships; no formal cutover | Cosmetic — no users to migrate |
| Embedded vs sidecar | Embedded only | Sidecar mode isn't needed and isn't requested |
| Hot reload of `router` config | **In-process mutations only.** The bridge HTTP handlers update the in-memory `cfg.Router` and call `config.UpdateCfgFile` in the same call (point (d) in Configuration). External edits to `.opencode.json` require a process restart — opencode does **not** currently watch its config file (verified: no `viper.WatchConfig`, no SIGHUP handler). A file watcher is explicitly out of scope; see Open Question #2 if we revisit. | One mutation path (HTTP). External-edit hot-reload would need new watcher infrastructure for the whole config, not just `router` |

## Configuration

The bridge configuration lives under a new top-level `router` key in `.opencode.json`. Only fields the bridge actually uses; nothing inherited from the TS bridge's `directory`/`opencodeUrl`/`opencodeDirectory` (all redundant in-process).

```json
{
  "data": { "directory": ".opencode" },
  "router": {
    "questionMode": "interactive",
    "permissionMode": "allow",
    "toolUpdatesEnabled": true,
    "channels": {
      "telegram": {
        "enabled": true,
        "bots": [
          {
            "id": "default",
            "token": "...",
            "enabled": true,
            "access": "private",
            "pairingCodeHash": "<sha256-hex>",
            "groupsEnabled": false
          }
        ]
      },
      "slack": {
        "enabled": true,
        "apps": [
          { "id": "default", "botToken": "xoxb-...", "appToken": "xapp-...", "enabled": true }
        ]
      },
      "mattermost": {
        "enabled": true,
        "instances": [
          { "id": "default", "serverUrl": "https://mm.example.com", "accessToken": "...", "enabled": true, "groupsEnabled": true }
        ]
      }
    }
  }
}
```

Field semantics match the TS bridge's existing `opencode-router.json` field-for-field (so the JSON shape is portable for anyone copy-pasting from old configs) except that:

- `directory`, `opencodeUrl`, `opencodeDirectory`, `opencodeUsername`, `opencodePassword` are **dropped** — the bridge is in-process, working directory is `config.WorkingDirectory()`.
- Per-identity `directory` is **dropped** for the same reason.
- Top-level `groupsEnabled` becomes a per-identity field only (the global fallback the TS bridge had is gone — set it on the identity you mean).

**Go type layout** (in `internal/bridge/config.go`):

```go
type Config struct {
    QuestionMode       string       `json:"questionMode"`        // "interactive" | "auto-reject" | "disabled"
    PermissionMode     string       `json:"permissionMode"`      // "allow" | "deny"
    ToolUpdatesEnabled bool         `json:"toolUpdatesEnabled"`
    Channels           ChannelsConf `json:"channels"`
}
```

Hooked into `internal/config/config.go`'s `Config` struct as `Router *bridge.Config \`json:"router,omitempty"\``. When `nil` or no channel enabled, the bridge does not start.

**HTTP mutations write back to `.opencode.json`.** The current write helper is the package-private `updateCfgFile(updateCfg func(*Config)) error` at `internal/config/config.go:1205-1254`. It has three deficiencies the Go bridge must address before depending on it for identity/token CRUD:

1. **Not atomic.** It calls plain `os.WriteFile`; a crash mid-write can leave a truncated/invalid JSON.
2. **Hard-coded `0o644` mode.** `.opencode.json` will hold Slack bot/app tokens, Mattermost personal access tokens, and Telegram bot tokens. World-readable mode leaks them to any local user on multi-user hosts or shared CI runners. Every CRUD call also clobbers operator-applied `chmod 600`.
3. **Does not refresh the in-memory `cfg` singleton.** It reads from disk, mutates the parsed copy, writes back — `config.cfg` (what `config.Get()` returns) is untouched. Existing callers (`UpdateTheme` at `config.go:1311`, `UpdateVimMode` at `config.go:1326`, `UpdateAgentModel` at `config.go:1274`) work around this by manually mutating `cfg.X` *before* invoking `updateCfgFile`.

Phase 1.2 ships the fix as one consolidated change:

- **(a) Export** it as `config.UpdateCfgFile`.
- **(b) Atomic write.** Replace the `os.WriteFile` call with: open `<configFile>.tmp` in the same directory (same-fs is required for atomic rename), write, `f.Sync()`, `f.Close()`, `os.Rename(tmp, target)`, then `fsync` the parent directory via `os.Open(parentDir)` + `f.Sync()`. The parent-dir fsync is what actually guarantees rename durability on POSIX — without it a power-loss crash can lose the rename even after fsyncing the file. On Windows, Go's `os.Rename` calls `MoveFileEx(MOVEFILE_REPLACE_EXISTING)`, which is best-effort atomic on NTFS but not formally guaranteed; for Win10+ callers wanting strict atomicity, switch to `golang.org/x/sys/windows`'s `SetFileInformationByHandle(FileRenameInfoEx, FILE_RENAME_FLAG_POSIX_SEMANTICS)`. Acceptable to ship best-effort on Windows initially — POSIX is the production target.
- **(c) Enforce safe file mode.** Always write the temp file with `0o600` when `cfg.Router` is non-nil and contains any token-bearing channel (Telegram `token`, Slack `botToken`/`appToken`, Mattermost `accessToken`). For configs without secrets, preserve existing mode via `os.Stat` (don't surprise operators who chmod'd otherwise). For new files, default `0o600`. The point is to *upgrade* pre-existing `0o644` installs that just gained tokens via identity CRUD — never silently inherit a lax mode the moment the file starts carrying secrets, and never silently widen one.
- **(d) Caller contract: live state.** `UpdateCfgFile` does not refresh `config.cfg`. Every caller — bridge HTTP handlers, existing `UpdateTheme`/`UpdateVimMode`/`UpdateAgentModel` — must mutate the in-memory `cfg` alongside the file write. The bridge orchestrator reads `cfg.Router` to start/stop adapters; without the live mutation, a `POST /identities/telegram` would update the file, return 200, and leave the running bridge with stale state until the next process restart. The contract is documented on `UpdateCfgFile`'s godoc.

### In-process integration points

```
┌──────────────────────────────────────────────────────────────┐
│ opencode serve                                               │
│                                                              │
│  ┌── chat WebSocket / long-poll ─────┐                       │
│  │  Telegram / Slack / Mattermost    │                       │
│  └────────────────┬──────────────────┘                       │
│                   │ inbound event                            │
│                   ▼                                          │
│           internal/bridge/bridge.go ── prompt ──►            │
│           ┌─────────────┐                                    │
│           │ Orchestrator│                                    │
│           └──────┬──────┘                                    │
│                  │                                           │
│      ┌───────────┴────────────┬─────────────────┐            │
│      ▼ direct call            ▼                 ▼            │
│  app.ActiveAgent()    messages.PartEvent     sessions.Get    │
│      .Run(...)        broker.Subscribe       /Create         │
│                       (no SSE/HTTP)                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Key wins vs. today:

1. `messages.PartEvent` broker subscribed via Go channel; no SSE parsing.
2. `agent.Run` invoked directly; no `session.prompt` HTTP call.
3. Permission/question responses delivered via `permission.Service` and `question.Service` method calls; no `POST /question/{id}/reply`.
4. The bridge's `/send` endpoint stays HTTP (external clients call it), wired into opencode's existing mux.

**Run / part-event interaction is not single-channel.** `agent.Run(ctx, sessionID, content, maxTurnsOverride, attachments...)` returns `(<-chan AgentEvent, error)` where the channel delivers one terminal `AgentEvent` and closes — it is **not** a stream of parts. Per-tool progress (the `[tool] pending|running|completed` lines the bridge surfaces today) arrives via a **separate** subscription to `messages.SubscribeParts(ctx)`. The orchestrator therefore subscribes parts **before** invoking `Run`, demuxes the resulting events by `sessionID`, and waits on the Run channel for completion.

**Why subscribe-before-Run is mandatory, not stylistic.** `PublishPart` (`internal/message/message.go:76-89`) has a zero-subscribers fast path: `if s.parts.GetSubscriberCount() == 0 { return }`. Events emitted before any subscriber attaches are dropped on the floor — they are **not** buffered for late subscribers. If the orchestrator calls `Run` first and then `SubscribeParts`, the agent's first `ToolCall pending` parts vanish silently with no error path. This is the load-bearing reason for the ordering; any refactor that swaps it will silently lose initial tool transitions.

**Broker is process-wide; demux must not do IO inline.** `SubscribeParts` returns a single channel carrying part events from **every session** in the process — TUI rendering, every active agent, the bridge. If the bridge's demux goroutine blocks (e.g., a slow `chat.postMessage` round-trip while still holding the receive loop), the broker's per-subscriber buffer fills and `PublishPart` back-pressures across all sessions, including the local TUI. Required pattern: the goroutine receiving from the broker channel drains immediately into a per-`sessionID` buffered queue (or per-session goroutine); **no outbound IO on the receive loop**. One wedged Mattermost connection must not stall TUI rendering.

**Per-session queue dimensioning and overflow.** Each per-`sessionID` queue is a buffered Go channel of size **64** — picked to cover a worst-case turn of ~12 parallel tool calls × 3 lifecycle transitions (`pending|running|completed`) plus a small headroom for text streaming, without committing significant memory per session. On overflow (the per-session sender goroutine can't accept fast enough — i.e., the chat platform is wedged), drop the **oldest** event and append the new one, then emit a single warn-level log per session per minute (`bridge: part-queue overflow session=<id> dropped=<n>`). Drop-oldest because newer transitions are strictly more informative (`completed` supersedes `running` supersedes `pending`); silently keeping the stale event would mislead the operator chat surface. The bridge demux goroutine itself NEVER blocks — its send to a per-session queue is a non-blocking `select { case q <- ev: default: dropAndLog(ev) }`. This guarantees a wedged adapter cannot back-pressure into the process-wide broker.

**Busy semantics.** `agent.Run` returns `ErrSessionBusy` when the session already has an in-flight request. The TS bridge today enqueues via `enqueue(key, fn)` (an in-memory channel-per-key serializer). The Go port keeps the same pattern in the orchestrator: one goroutine per `(channel, identity, peerKey)`, requests serialized through a buffered channel.

### Storage: dual provider (SQLite + MySQL)

Opencode supports two session providers selected via `config.SessionProvider.Type` (`internal/db/provider.go:28-42`). Both are first-class — MySQL is the primary deploy target; SQLite is the local-dev default. The bridge port must support both.

Concretely, what already exists:

- **`internal/db/migrations/sqlite/`** and **`internal/db/migrations/mysql/`** — parallel goose migrations.
- **`internal/db/sql/*.sql`** (SQLite query files) and **`internal/db/sql/mysql/*.sql`** (MySQL query files) — sqlc generates Go code into `internal/db/` (package `db`) and `internal/db/mysql/` (package `mysqldb`) respectively. See `sqlc.yaml`.
- **`internal/db/querier_factory.go`** — runtime selector between the two generated querier packages.
- **`internal/db/embed.go`** already embeds both migration directories.

The bridge port follows the same pattern. Every table gets two migrations + two query files + sqlc regenerates both packages. The bridge's store layer uses the existing factory to talk to whichever provider is configured.

**Project scoping.** Opencode sessions carry `project_id` (added in `20260131204842_add_project_id.sql`) so that a shared MySQL can host multiple opencode workspaces without collision. The bridge tables inherit the same constraint — every row carries `project_id` and PKs include it. Two opencode instances against the same MySQL with different workspaces will not collide on `(channel, identity_id, peer_id)` because the actual key is `(project_id, channel, identity_id, peer_id)`. The `project_id` value is computed via `db.GetProjectID(workingDir)` — same as opencode's own sessions.

### Storage layout

```
opencode database (SQLite file or MySQL schema)
├── sessions, messages, ...            ← existing opencode tables (have project_id)
├── bridge_sessions                    ← NEW: peer → opencode session_id pointer
│     project_id, channel, identity_id, peer_id, session_id,
│     created_at, updated_at
│     PK (project_id, channel, identity_id, peer_id)
│     FK session_id → sessions(id) ON DELETE SET NULL
└── bridge_allowlist                   ← NEW: per-identity peer allowlist
      project_id, channel, identity_id, peer_id, created_at
      PK (project_id, channel, identity_id, peer_id)
```

Two tables, no more. The TS bridge's `bindings` is unnecessary (workspace directory is `config.WorkingDirectory()`); its `settings` table is unnecessary (settings live in `.opencode.json` `router` section, persisted via `config.UpdateCfgFile`).

`project_id` is in every PK so a shared MySQL can host multiple opencode workspaces without collision. `project_id` value is `db.GetProjectID(config.WorkingDirectory())` — same source opencode uses for `sessions.project_id`.

**`FK ON DELETE SET NULL` on session_id**: when opencode garbage-collects a session, the bridge's pointer becomes `NULL` (rather than CASCADE-deleting the row). On the next inbound message from that peer the bridge sees `session_id IS NULL`, creates a fresh opencode session, and updates the pointer.

**Indexes** (in addition to PKs):

| Table | Index | Reason |
|---|---|---|
| `bridge_sessions` | `idx_bridge_sessions_session_id (session_id)` | The FK `ON DELETE SET NULL` requires MySQL/SQLite to scan child rows when a parent `sessions` row is deleted. Without this index, deleting an opencode session triggers a full-scan of `bridge_sessions`. |
| `bridge_allowlist` | covered by PK | Lookups are exact-key `(project_id, channel, identity_id, peer_id)`. |

| Engine | File | Concerns |

| MySQL  | `internal/db/migrations/mysql/<ts>_add_bridge_tables.sql` | `VARCHAR(N)`, `BIGINT`, `UNIX_TIMESTAMP()` defaults, `BEFORE UPDATE` trigger, `ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci`, inline `CONSTRAINT fk_bridge_sessions_session FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE SET NULL` |

**sqlc query files to ship**:

- `internal/db/sql/bridge_sessions.sql` — `:one` with `RETURNING *`, plus `:exec` for update/delete.
- `internal/db/sql/mysql/bridge_sessions.sql` — `:execresult` for INSERT (no RETURNING), then GET-after-INSERT pattern (same as existing MySQL sessions query at `internal/db/sql/mysql/sessions.sql:1-33`).
- Same pair for `bridge_allowlist`.

Re-run sqlc after editing the schema or query files: `make generate` (Makefile target wraps `go generate ./...`). The bridge store wrapper mirrors `querier_factory.go` style, multiplexing between `db.Queries` and `mysqldb.Queries` for the two bridge tables.

**sqlc input layout** (must update all three for MySQL):

1. `internal/db/schema/mysql.sql` — append the two bridge table DDLs in MySQL syntax (this is the file sqlc compiles against).
2. `internal/db/sql/mysql/bridge_*.sql` — query files.
3. `internal/db/migrations/mysql/<ts>_add_bridge_tables.sql` — runtime migration applied by goose. (Same DDL as #1, but in goose-annotated form.)

For SQLite, sqlc's schema source IS the migration directory (`sqlc.yaml:5` → `schema: "internal/db/migrations/sqlite"`), so #1 and #3 collapse to a single file.

**Not in SQLite/MySQL**: Telegram pairing hashes live in `.opencode.json` (`router.channels.telegram.bots[].pairingCodeHash`). Not state — pure config.

### No data migration

No data migration tool ships with this port. There is no production traffic to preserve, no existing chat sessions to carry over, no `~/.openwork/opencode-router/opencode-router.db` to import from. Operators starting fresh paste their tokens into `.opencode.json`'s new `router` section (or POST them to the identity endpoints) and the bridge comes up with empty tables.

## Edge Cases

### Bridge config invalid at startup

A misconfigured `router` section in `.opencode.json` must not block the rest of `opencode serve` from coming up. Decisions:

1. `router` schema validation error (unknown channel kind, missing required field on an identity) → opencode boots normally; bridge logs the error once and exposes it via `/health` → `bridge: {status: "error", error: "..."}`.
2. Per-identity startup failure (bad Telegram token, Slack auth rejected) → adapter for that identity is marked disabled; other identities continue. `/health` reports `bridge.adapters[id].status` per identity.
3. Missing `router` section (or no enabled channel) → bridge is silently disabled. Equivalent to "opt-out".

Operators differentiate "API down" from "bridge down" by polling `/health`. Single failure domain (panic in a bridge goroutine) cannot take down the API — every adapter goroutine and the orchestrator's run handler wraps work in `defer recover()`.

### Two `opencode serve` instances pointing at the same database

Concurrent processes can race three ways:

1. **Database writes** — serialized by SQLite's WAL or MySQL's row locks. Safe.
2. **WebSocket/long-poll against the same chat-platform credentials** — duplicate inbound delivery and racing run dispatch. **Not safe.** Telegram's `getUpdates` long-poll uses an offset that both processes will trample; Mattermost's WebSocket will deliver the same `posted` event to both connections.
3. **`/agent/select` and other mutating endpoints** — last writer wins; harmless beyond user confusion.

Mitigation for #2 depends on the provider:

- **SQLite (local-dev)**: at startup the bridge takes an OS file lock on `<config.Data.Directory>/bridge.lock` via the new `internal/fileutil/lock.go` (hand-rolled — `syscall.Flock` on Unix, `LockFileEx` on Windows; ~60 LOC total, no third-party dep). If held, the bridge refuses to start. Released automatically on process exit. Suffices because SQLite is always per-host.
- **MySQL (primary deploy)**: scoped per-identity. For each enabled identity, the bridge acquires a MySQL named lock via `GET_LOCK(?, 0)` where the name is `'opencode_bridge:' + SHA1_HEX(project_id + ':' + channel + ':' + identity_id)`. The SHA1 keeps the name under MySQL's **64-character** lock-name limit (16 + 40 = 56 chars). `project_id` (derived from `db.GetProjectID(WorkingDirectory())`) is part of the hash because `GET_LOCK` names are **server-wide and case-insensitive** — they are not scoped to a schema. Two opencode deployments against the same MySQL server in different schemas would otherwise collide on `'opencode_bridge:slack:default'`; hashing project_id in keeps them disjoint as long as their workspace paths differ. (Two operators using the exact same workspace path against the same MySQL server is the genuinely ambiguous case and is what the lock is *supposed* to prevent.) Two implementation constraints the implementer must honor:
  1. **Dedicated connection per identity.** `GET_LOCK` is per-connection; releasing the connection to a pool releases the lock. The bridge must hold a dedicated `*sql.Conn` (via `db.Conn(ctx)`) for the lifetime of each adapter, never returning it to the pool.
  2. **Reacquire on disconnect.** If the dedicated connection drops (network blip, MySQL restart), the bridge must observe via ping failure, reconnect, and re-acquire the lock before resuming the adapter. Until reacquired the adapter is marked degraded in `/health`.

  If initial acquisition fails (another instance owns the lock), the adapter is marked disabled with a clear "another opencode instance owns this identity" error in `/health`. Other identities on the same process come up normally. This is the right granularity: an operator running two opencode instances on the same MySQL with the same project_id can split identities between them.

Documented in DEPLOY.md as "single-writer per identity, enforced by the database". Operators who genuinely want HA need to split the config (different identity_ids per instance) — true active-active requires solving the chat-platform-side single-consumer constraint, which is out of scope.

### Bridge crashes a chat session, opencode keeps running

In the embedded model, a panic in the bridge could take down `opencode serve`. Mitigation: every adapter goroutine and the orchestrator's run-handler wraps work in `defer recover()` and logs the panic without propagating.

### Concurrent `/agent/select` from multiple chats

Already a pre-existing race in `app.SetActiveAgent`. Spec for agent/model selection acknowledges it. Embedded mode doesn't change the race window; just makes it more obvious.

### Slack Socket Mode handshake under flaky network

The TS implementation has retry logic with backoff; `slack-go/slack` Socket Mode handles this natively. Port should rely on the lib's behavior rather than re-implementing.

### Telegram long-poll vs webhook

TS uses long-poll. Go port stays on long-poll — no webhook URL required matches the bridge's "outbound-only" deployment story.

### Media files larger than chat platform limit

TS bridge returns an error to the agent. Go bridge: same. Limits per platform (Telegram 50 MB, Slack 1 GB, Mattermost configurable) should be enforced at outbound time.

### Session-scoped behavior under bridge restart

Today: bridge restarts, in-flight runs reconnect to SSE on the next inbound message. Embedded: bridge restart = opencode restart = in-flight runs already lost. Acceptable — opencode itself owns the run; if it dies, the run dies regardless of bridge architecture.

## Open Questions

1. **Telegram library choice — `go-telegram/bot` vs `telegram-bot-api/v5`?**
   - `go-telegram/bot` has a cleaner mid-2020s API but ~3k stars; `v5` has ~5k stars and is the older default.
   - **Recommendation**: `go-telegram/bot`. The grammy API in TS is closer in spirit. If we hit feature gaps (rare), switch.

2. **Should we add a file watcher to `.opencode.json` for external edits?**
   - Opencode does **not** watch its config file today (no `viper.WatchConfig` / SIGHUP / fsnotify on the config path — fsnotify is only used in `internal/lsp/watcher/`). HTTP-mutation hot-reload is handled in-process per Configuration point (d); the open question is whether *external* edits (operator opens `.opencode.json` in vim and saves) should trigger live adapter restart.
   - **Recommendation**: out of scope for this port. The dashboard / HTTP API are the canonical mutation surface; external edits requiring a restart matches every other section of `.opencode.json` today. Revisit only if operator demand surfaces.

3. **Test framework for the port**
   - Go std `testing` with `httptest` for HTTP-shaped contracts; `gorilla/websocket` for WebSocket mocking (Mattermost adapter).
   - **Recommendation**: std `testing` + `gorilla/websocket`. No new test framework dependency.
